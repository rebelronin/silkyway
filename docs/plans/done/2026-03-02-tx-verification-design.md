# Transaction Verification Design

**Date:** 2026-03-02
**Scope:** `@silkysquad/silk` SDK — new `verify` module

## Problem

Agents request transactions from the SilkyWay backend, receive unsigned base64-encoded Solana transactions, and sign them. There is currently no tooling for an agent to verify that the transaction it received matches what it requested, or to audit all instructions in the transaction before signing.

## Goals

- **Primary (A):** Confirm our backend built exactly the transaction the agent requested (correct program, accounts, amounts).
- **Secondary (C):** Give agents a full breakdown of every instruction in any Solana transaction, including instructions from third-party programs, so unexpected or suspicious instructions can be flagged.

## Non-goals

- Adversarial/MITM protection (out of scope for now).
- Verification of transactions not generated by us (supported generically, but not the primary use case).
- Config file integrity / tamper-proofing (future concern).

## Design

### API

Two functions exported from `@silkysquad/silk`:

```typescript
// General-purpose: "what does this transaction do?"
const analysis = await silk.analyzeTransaction(txBase64, { connection?, config? });

// Intent-check: "did the backend do what I asked?"
const result = await silk.verifyIntent(txBase64, intent, { connection?, config? });
```

#### `analyzeTransaction` output

```typescript
interface TransactionAnalysis {
  feePayer: string;
  instructions: InstructionAnalysis[];
  flags: RiskFlag[];
  summary: string; // LLM-friendly plain English
}

interface InstructionAnalysis {
  index: number;
  programId: string;
  programName: string | null;       // "Handshake", "SPL Token", null if unknown
  type: string | null;              // "create_transfer", "transfer", null if unknown
  known: boolean;
  params: Record<string, unknown>;  // decoded, human-readable values
  flags: RiskFlag[];
}

interface RiskFlag {
  severity: 'info' | 'warning' | 'error';
  code: string;
  message: string;
  instructionIndex?: number;
}
```

#### `verifyIntent` output

```typescript
interface VerifyResult {
  verified: boolean;
  discrepancies: string[];  // human-readable, e.g. "Amount mismatch: expected 100 USDC, got 50 USDC"
  analysis: TransactionAnalysis;
}
```

Intent shape mirrors existing backend request params:

```typescript
type Intent =
  | { type: 'create_transfer'; sender: string; recipient: string; amount: number; token?: string; mint?: string; memo?: string }
  | { type: 'claim_transfer'; claimer: string; transferPda: string }
  | { type: 'cancel_transfer'; canceller: string; transferPda: string }
  | { type: 'transfer_from_account'; owner: string; recipient: string; amount: number; token?: string }
  | { type: 'deposit'; owner: string; amount: number; token?: string };
```

### Local Registry

The SDK ships a bundled `registry.json` with two lookup tables. This allows most analysis to run without RPC calls.

**Programs:**
```json
{
  "11111111111111111111111111111111": { "name": "System Program", "decoder": "system" },
  "TokenkegQfeZyiNwAJbNbGKPFXCWuBvf9Ss623VQ5DA": { "name": "SPL Token", "decoder": "spl-token" },
  "ATokenGPvbdGVxr1b2hvZbsiqW5xWH25efTNsLJe1bC5": { "name": "Associated Token Account", "decoder": "ata" },
  "HANDu9uNdnraNbcueGfXhd3UPu6BXfQroKAsSxFhPXEQ": { "name": "Handshake", "decoder": "handshake" },
  "SiLKos3MCFggwLsjSeuRiCdcs2MLoJNwq59XwTvEwcS": { "name": "Silkysig", "decoder": "silkysig" },
  "JUP6LkbZbjS1jKKwapdHNy74zcZ3tLUZoi5QNyVTaV4": { "name": "Jupiter v6", "decoder": "jupiter" }
}
```

**Tokens:**
```json
{
  "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v": { "symbol": "USDC", "decimals": 6 },
  "So11111111111111111111111111111111111111112": { "symbol": "SOL", "decimals": 9 },
  "Es9vMFrzaCERmJfrF4H2FYD4KCoNkY11McCe8BenwNYB": { "symbol": "USDT", "decimals": 6 }
}
```

Unknown programs are flagged as `UNKNOWN_PROGRAM` — no RPC lookup attempted, as unknown programs are inherently suspicious. Unknown token mints fall back to RPC metadata fetch.

### Decoders

Each decoder accepts raw instruction bytes + accounts array and returns a typed `params` object.

| Decoder | Approach |
|---|---|
| `system` | Hardcoded (System Program layout is stable) |
| `spl-token` | Hardcoded (SPL Token layout is stable) |
| `ata` | Hardcoded |
| `handshake` | IDL-based (uses bundled `handshake-idl.json`) |
| `silkysig` | IDL-based (uses bundled `silkysig-idl.json`) |
| `jupiter` | Best-effort (Jupiter route instructions are complex; decodes what it can) |

### Risk Flagging Rules

Rules run automatically on every `analyzeTransaction` call:

| Code | Severity | Trigger |
|---|---|---|
| `UNKNOWN_PROGRAM` | error | Any instruction from a program not in the registry |
| `UNEXPECTED_SOL_DRAIN` | error | System Program transfer to an address that isn't the fee payer or a known program |
| `UNEXPECTED_TOKEN_TRANSFER` | warning | SPL Token transfer to an address not involved in the requested intent |
| `MULTIPLE_FEE_PAYERS` | warning | Fee payer differs from the signing agent wallet |
| `LARGE_COMPUTE_BUDGET` | info | SetComputeUnitPrice instruction present |

### `verifyIntent` Logic

For each intent type, `verifyIntent`:
1. Calls `analyzeTransaction` to get the full breakdown.
2. Finds the expected instruction by program + type.
3. Field-compares decoded params against the intent (amount within 0.01% tolerance for floating point).
4. Fails if any `error`-severity flag exists on any instruction.
5. Returns `verified: false` with human-readable `discrepancies[]` on any mismatch.

### File Layout (in `silk` repo)

```
silk/src/verify/
  index.ts          ← exports analyzeTransaction, verifyIntent
  registry.json     ← bundled program + token lookup tables
  decoders/
    system.ts
    spl-token.ts
    ata.ts
    handshake.ts
    silkysig.ts
    jupiter.ts
  flags.ts          ← risk rule engine
  rpc.ts            ← RPC enrichment (unknown token metadata)
```

## Typical Agent Usage

```typescript
import { silk } from '@silkysquad/silk';

const { transaction, transferPda } = await backend.createTransfer({
  sender: agentWallet, recipient: bob, amount: 100, token: 'USDC',
});

const result = await silk.verifyIntent(transaction, {
  type: 'create_transfer',
  sender: agentWallet,
  recipient: bob,
  amount: 100,
  token: 'USDC',
}, { connection });

if (!result.verified) {
  throw new Error(`Transaction verification failed: ${result.discrepancies.join(', ')}`);
}

const signed = agentKeypair.sign(transaction);
await backend.submitTx(signed);
```

## Future Considerations

- Config file integrity verification (checksum or signed manifest) to prevent local registry tampering.
- Extending the registry with more DeFi protocols (Orca, Raydium, Drift, etc.).
- A `silk verify <tx-base64>` CLI command for manual inspection during development.
